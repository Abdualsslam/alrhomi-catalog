# ورقة اختبار فني Full-Stack  
## مشروع: Alrhomi Catalog

## مقدمة

هذا الاختبار مبني على مشروع قائم بالفعل، والهدف منه ليس إعادة بناء المشروع أو إدخال تغييرات معمارية كبيرة، بل فهم النظام الحالي ثم تنفيذ مجموعة تحسينات وتطويرات عملية عليه بشكل منظم ونظيف.

المطلوب هنا أن تتعاملي مع المشروع كعضوة فريق تستلم كود موجود مسبقًا، وتفهمينه، ثم تطورين عليه بدون كسر السلوك الحالي أو إدخال تعقيد غير ضروري.

المشروع يحتوي على:
- Backend مبني بـ NestJS
- Frontend مبني بـ React + TypeScript + MUI
- MongoDB
- Redis / Bull Queue
- إدارة منتجات وفئات وصور ولوحة تحكم

---

## الهدف من الاختبار

نريد تقييمك في النقاط التالية:

- قدرتك على فهم مشروع موجود مسبقًا
- قدرتك على العمل Full-Stack
- جودة قراراتك التقنية
- نظافة الكود
- احترامك لهيكل المشروع الحالي
- اهتمامك بالحالات الطرفية وتجربة الاستخدام
- طريقتك في تقسيم العمل ووضوحك في التسليم
- التزامك بأسلوب عمل احترافي على GitHub

## آلية العمل المطلوبة على GitHub

يجب تنفيذ الاختبار على **Branch منفصل** وعدم العمل مباشرة على `main` أو `develop`. الفكرة من الـ branches أنها تسمح بتطوير ميزة أو إصلاح بشكل معزول وآمن بدون التأثير على الفرع الرئيسي، ثم يتم اقتراح الدمج عبر Pull Request لمراجعة التغييرات قبل دمجها. ([docs.github.com](https://docs.github.com/articles/about-branches))

### اسم البرانش المطلوب

استخدمي هذا الاسم:

```bash
test/alrhomi-catalog-<your-name>
```

مثال:
```bash
test/alrhomi-catalog-sara
```

إذا رغبتِ في تقسيم العمل داخليًا إلى أكثر من جزء داخل نفس البرانش فهذا مسموح، لكن **التسليم النهائي يكون من خلال نفس البرانش**.

### المطلوب بخصوص GitHub workflow

- اسحبي آخر نسخة من الفرع الأساسي المعتمد في المشروع
- أنشئي Branch جديدًا بالاسم المذكور
- نفذي العمل عليه
- ارفعي التعديلات إلى GitHub
- افتحي Pull Request واضحًا ومنظمًا للمراجعة

في GitHub، الـ Pull Request هو الآلية الأساسية لاقتراح التغييرات ومناقشتها ومراجعتها قبل الدمج، وهذا جزء مهم من التعاون داخل الفرق. ([docs.github.com](https://docs.github.com/articles/creating-a-pull-request))

## أسلوب الـ Commit المطلوب

نريد History نظيفًا ومفهومًا، لذلك **لا نريد commitات عشوائية** مثل:
- update
- fix
- work
- final
- done
- test

نريد رسائل commit واضحة ومهنية.

### الصيغة المطلوبة

استخدمي أسلوبًا قريبًا من **Conventional Commits**، والذي يعتمد على الصيغة التالية:

```bash
type(scope): short description
```

وهذا الأسلوب يساعد على جعل سجل التعديلات واضحًا وسهل القراءة. ([conventionalcommits.org](https://www.conventionalcommits.org/en/v1.0.0/))

### الأنواع المقترحة

- `feat`: عند إضافة ميزة جديدة
- `fix`: عند إصلاح خطأ
- `refactor`: عند تحسين الكود بدون تغيير سلوك واضح
- `chore`: لأعمال تنظيمية أو تقنية بسيطة
- `docs`: للتوثيق
- `style`: لتنسيق أو تحسين شكلي فقط
- `test`: عند إضافة أو تعديل اختبارات

### أمثلة جيدة

```bash
feat(products): add filter for products without images
fix(categories): prevent deleting category linked to products
feat(catalog): sync filters with URL query params
fix(folders): protect admin folder routes with auth guard
refactor(health): improve readiness check for mongo and redis
docs(test): add implementation notes for delivered tasks
```

### أمثلة غير جيدة

```bash
fix
update code
final changes
work on backend
new edits
done
```

### عدد الـ commits

- نفضل أن تكون الـ commits **منطقية ومقسمة حسب الإنجاز**
- لا نريد commit واحد ضخم جدًا لكل العمل
- ولا نريد أيضًا عددًا كبيرًا جدًا من commits الصغيرة غير المفيدة
- الأفضل أن يكون كل commit يمثل خطوة واضحة ومفهومة

### مهم جدًا

إذا نفذتِ أكثر من شيء مختلف، فحاولي الفصل بينها بقدر معقول.  
مثلاً:
- commit للحماية
- commit لفلتر المنتجات بدون صور
- commit لمزامنة الفلاتر مع الرابط
- commit لتحسين حالات التحميل والخطأ

هذا يعطينا صورة أوضح عن طريقة تفكيرك وتنظيمك.

## أسلوب الـ Pull Request المطلوب

بعد الانتهاء، افتحي Pull Request من البرانش الخاص بك إلى الفرع الأساسي المعتمد في المشروع.

### عنوان الـ PR المقترح

```bash
[TEST] Alrhomi Catalog Improvements - <Your Name>
```

مثال:
```bash
[TEST] Alrhomi Catalog Improvements - Sara
```

### وصف الـ PR يجب أن يحتوي على الآتي

#### 1) Summary
ملخص قصير لما تم تنفيذه.

#### 2) Implemented Tasks
اذكري فيه المهام التي أنجزتِها.

#### 3) Notes / Assumptions
اذكري أي افتراضات بنيتِ عليها التنفيذ.

#### 4) Pending / Limitations
اذكري أي جزء لم يكتمل أو أي ملاحظة مهمة.

#### 5) Testing Notes
اذكري كيف اختبرتِ العمل بشكل مختصر.

### قالب مقترح للـ PR

```md
## Summary
Implemented the required full-stack improvements for the Alrhomi Catalog project.

## Implemented Tasks
- Protected folder management routes
- Improved readiness check for MongoDB and Redis
- Prevented deleting categories linked to products
- Added "products without images" filter in admin
- Synced catalog filters with URL
- Improved loading, error, and empty states in key pages

## Notes / Assumptions
- Assumed that folder management is admin-only
- Assumed that products with empty image arrays are considered "without images"

## Pending / Limitations
- No extra architectural changes were introduced
- Optional enhancements were kept minimal

## Testing Notes
- Tested catalog filter sync with refresh and browser navigation
- Tested category delete validation
- Tested admin filter for products with and without images
```

# المطلوب العام

نفّذي المهام التالية مع الالتزام بما يلي:

- لا تغيّري المعمارية العامة للمشروع
- لا تدخلي مكتبات جديدة إلا عند الحاجة الواضحة
- حافظي على نفس أسلوب المشروع الحالي قدر الإمكان
- أي API جديدة أو تعديل في API يجب أن يكون واضحًا ومنظمًا
- أي تطوير في الواجهة يجب أن يكون بسيطًا وواضحًا ويخدم الاستخدام الفعلي
- اهتمي بالرسائل، والحالات الفارغة، والأخطاء، وسهولة الاستخدام
- إذا وجدتِ تضاربًا بسيطًا في الأنواع أو البيانات بين الفرونت والباك، عالجيه بشكل نظيف

# المهام الأساسية

## أولًا: مهام Back-End

### 1) حماية مسارات المجلدات
يوجد جزء خاص بالمجلدات داخل النظام. المطلوب مراجعة هذا الجزء والتأكد من أن العمليات الحساسة محمية بشكل مناسب.

### المطلوب
- حماية عمليات إنشاء المجلد
- حماية إعادة التسمية
- حماية حذف المجلد
- توضيح هل عرض محتويات المجلد يجب أن يكون محميًا أم لا، ثم تطبيق القرار بشكل منطقي

### معايير القبول
- لا تستطيع أي جهة غير مصرح لها تنفيذ العمليات الإدارية
- السلوك متسق مع بقية النظام
- لا يحدث كسر في الواجهة بعد تطبيق الحماية

### تلميح
راجعي طريقة حماية مسارات الإدارة الأخرى في المشروع، ولا تبتكري أسلوبًا جديدًا مختلفًا عن الموجود.

---

### 2) تحسين readiness check
يوجد health check في المشروع، لكن readiness حاليًا بسيط جدًا.

### المطلوب
اجعلي endpoint الجاهزية يتحقق فعليًا من:
- MongoDB
- Redis

### معايير القبول
- إذا كان كل شيء يعمل، يعيد readiness بشكل صحيح
- إذا كانت إحدى الخدمات الأساسية غير متاحة، تظهر النتيجة بشكل أوضح
- الكود يكون بسيطًا ومفهومًا

### تلميح
لا نريد نظام monitoring معقدًا. فقط readiness حقيقي ومفيد للتشغيل.

---

### 3) منع حذف الفئة إذا كانت مرتبطة بمنتجات
حاليًا يوجد فحص لحالات معينة عند حذف الفئة، لكن المطلوب هنا التأكد من عدم حذف فئة مرتبطة بمنتجات.

### المطلوب
- قبل حذف الفئة، تحققي هل توجد منتجات مرتبطة بها
- إذا كانت مرتبطة، امنعي الحذف وأعيدي رسالة واضحة

### معايير القبول
- لا يمكن حذف فئة مرتبطة بمنتجات
- رسالة الخطأ مفهومة
- لا يتم حذف أي بيانات جزئيًا أو بشكل غير متوقع

### تلميح
انتبهي للعلاقة بين Category و Product، وفكري هل يكفي فحص category فقط أو هناك ما يجب مراعاته أيضًا.

## ثانيًا: مهام Front-End

### 4) مزامنة فلاتر الكتالوج مع رابط الصفحة
في صفحة الكتالوج، المطلوب أن تكون الفلاتر جزءًا من رابط الصفحة.

### المطلوب
مزامنة الفلاتر التالية مع الـ URL:
- البحث
- الفئة
- الموديل
- الصفحة الحالية

بحيث:
- عند تحديث الصفحة لا تضيع الفلاتر
- يمكن مشاركة الرابط بنفس النتائج
- يعمل الرجوع والتقدم في المتصفح بشكل طبيعي

### معايير القبول
- الرابط يعكس حالة الفلاتر
- الصفحة تقرأ القيم من الرابط عند الفتح
- أي تغيير في الفلاتر ينعكس على الرابط
- لا يحدث تضارب أو loop في state

### تلميح
نفذيها ببساطة. لا تبالغي في abstraction أو custom hooks إذا لم تكن هناك حاجة.

---

### 5) تحسين حالات التحميل والخطأ والحالة الفارغة
راجعي الصفحات الأساسية وحسّني تجربة المستخدم فيها.

### المطلوب
تحسين الحالات التالية على الأقل في الصفحات الأساسية:
- Loading
- Error
- Empty State

والأماكن المقترحة:
- الكتالوج
- الفئات
- صفحة إدارة المنتجات أو الصور

### معايير القبول
- لا ترى المستخدمة شاشة فارغة بدون تفسير
- رسائل الخطأ مفهومة
- الحالة الفارغة لها معنى واضح
- التحميل يظهر بشكل جيد

### تلميح
الهدف هنا ليس فقط الجمال، بل الوضوح والانسيابية.

---

## ثالثًا: مهام Full-Stack

### 6) إضافة فلتر “المنتجات بدون صور” في لوحة الإدارة
هذه المهمة تعتبر المهمة الأساسية المشتركة بين الفرونت والباك.

### المطلوب
إضافة فلتر في صفحة إدارة المنتجات لعرض:
- كل المنتجات
- المنتجات التي لديها صور
- المنتجات التي لا تحتوي على صور

مع إظهار عداد واضح للمنتجات بدون صور.

### المطلوب في الباك
- دعم فلترة المنتجات حسب وجود صور أو عدم وجودها

### المطلوب في الفرونت
- إضافة واجهة واضحة للفلاتر
- عرض النتائج بشكل صحيح
- إظهار عداد المنتجات بدون صور

### معايير القبول
- الفلتر يعمل فعليًا من الباك
- النتائج صحيحة
- العداد صحيح
- لا يؤثر سلبًا على الفلاتر الحالية أو pagination

### تلميح
هذه المهمة مهمة جدًا لأنها تكشف فهمك للعلاقات بين المنتجات والصور، وكذلك طريقة ربط الواجهة مع الـ API.

# مهام تحسين إضافية اختيارية

هذه ليست إلزامية، لكنها ستعطي انطباعًا ممتازًا إذا أُنجزت بشكل جيد.

## 7) إضافة ترتيب للمنتجات في الكتالوج
مثل:
- الأحدث
- الأقدم
- الاسم تصاعديًا
- الاسم تنازليًا

## 8) إضافة زر نسخ رمز المنتج
في صفحة الإدارة أو تفاصيل المنتج.

## 9) تحسين بسيط لصفحة تفاصيل المنتج
مثل:
- عرض أفضل للصور
- اختيار الصورة الرئيسية
- تحسين عرض المنتجات المشابهة

## 10) مراجعة التوافق بين أنواع الفرونت والباك
إذا وجدتِ اختلافات واضحة في الـ types أو response shapes، عالجيها بشكل نظيف وهادئ بدون مبالغة.

# ما الذي ننتظره في التسليم

## المطلوب تسليمه
- الكود كاملًا على البرانش المطلوب
- Pull Request واضح ومنظم
- شرح مختصر لما تم تنفيذه
- توضيح أي افتراضات اتخذتِها
- توضيح أي نقطة لم تكتمل إن وجدت

## داخل التقرير اذكري:
- ما الذي تم
- ما الذي لم يتم
- لماذا اخترتِ هذا الأسلوب
- هل واجهتِ أي مشكلة في المشروع الأصلي
- أي اقتراحات إضافية لاحظتِها أثناء العمل

# معايير التقييم

سيتم التقييم بناءً على التالي:

## 1) فهم المشروع
هل فهمتِ العلاقات والمنطق الحالي، أم تعاملتِ مع المشروع بشكل عشوائي.

## 2) جودة التنفيذ
هل الحلول عملية وواضحة، أم فيها تعقيد زائد.

## 3) نظافة الكود
هل الكود مقروء، منظم، ومتسق مع المشروع.

## 4) احترام الهيكل الحالي
هل التزمتِ بطريقة المشروع الحالية، أم أدخلتِ نمطًا مختلفًا تمامًا.

## 5) التعامل مع الحالات الطرفية
مثل:
- عدم وجود بيانات
- خطأ في الطلب
- فلتر بلا نتائج
- حذف عنصر مرتبط بعناصر أخرى

## 6) جودة تجربة المستخدم
هل التطويرات مفيدة فعلًا من ناحية الاستخدام.

## 7) جودة التسليم
هل الشرح واضح، وهل أوضحتِ ما فعلتِه بشكل مهني.

## 8) جودة العمل على GitHub
هل البرانش واضح  
هل الـ commits نظيفة  
هل الـ PR منظم ويشرح العمل بشكل جيد

# ملاحظات مهمة

- لا نريد over-engineering
- لا نريد إعادة كتابة أجزاء كاملة بلا داع
- نفضل الحل الأبسط الصحيح على الحل الأعقد
- لو وجدتِ فرصة تحسين صغيرة ومفيدة، يمكنك تنفيذها أو الإشارة إليها
- إذا اكتشفتِ مشكلة أصلية في المشروع، وثقيها بوضوح بدل تجاوزها بصمت
- لا تعملي مباشرة على الفرع الرئيسي
- لا ترفعي PR غير مرتب أو بدون وصف واضح
- لا تستخدمي رسائل commit عامة وغير احترافية

# تلميحات عامة

- ابدئي من الموجود ولا تبدئي من الصفر
- تتبعي أمثلة مشابهة داخل المشروع قبل كتابة أي جزء جديد
- إذا وجدتِ نفس النمط في جزء آخر من النظام، أعيدي استخدامه
- في الـ backend ركزي على:
  - validation
  - guards
  - رسائل الخطأ
  - consistency
- في الـ frontend ركزي على:
  - وضوح الحالة
  - سهولة الاستخدام
  - عدم تكرار الكود بلا حاجة
  - الربط الصحيح مع الـ API

# مصادر التعلم والاطلاع

لفهم فكرة الـ branches على GitHub، راجعي التوثيق الرسمي الذي يوضح أن الـ branch هو نسخة موازية من المستودع تسمح بالعمل بشكل معزول دون التأثير على الفرع الرئيسي.  
- https://docs.github.com/articles/about-branches

لفهم Pull Requests وطريقة إنشائها، راجعي توثيق GitHub الرسمي حول إنشاء الـ PR، وكذلك صفحة Pull Requests العامة التي تشرح أنها آلية اقتراح ومراجعة التغييرات قبل الدمج.  
- https://docs.github.com/articles/creating-a-pull-request

لفهم أسلوب كتابة الـ commits بشكل أوضح، راجعي **Conventional Commits**، وهو مرجع بسيط وممتاز لتنظيم رسائل الـ commit بصيغة مثل `feat(scope): description` أو `fix(scope): description`.  
- https://www.conventionalcommits.org/en/v1.0.0/

إذا كان المستودع عندكم يستخدم حماية للفروع أو مراجعات إلزامية قبل الدمج، فتوثيق GitHub الخاص بـ protected branches مفيد أيضًا لفهم هذا الأسلوب داخل الفرق.  
- https://docs.github.com/repositories/configuring-branches-and-merges-in-your-repository/managing-protected-branches/about-protected-branches

وبخصوص الجوانب التقنية للمشروع نفسه، فهذه أفضل مراجع مباشرة:
